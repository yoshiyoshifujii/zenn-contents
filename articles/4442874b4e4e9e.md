ソフトウェアを書くとき、私たちは自然と「どのようにデータを扱うか」を考えています。
しかし、その"考え方のレンズ"は、手続き的・オブジェクト指向・関数型といったパラダイムによって大きく異なるかなと思います。

この記事では、3つのパラダイムを「何に焦点を当てているか」という観点で整理してみます。

## 手続き的：状態と手順を、順番に積み上げる

手続き的思考は、コンピュータに対して「何を、どの順番で実行するか」を指示する考え方かなーと思います。

典型的なイメージはこんなコードです。

```go
conn := open()
tx := conn.Begin()
item := selectItem()
update(item)
tx.Commit()
```

ここでは「状態（state）」と、その状態をどう変化させるかが中心テーマになります。
仕事の工程表を作るような感覚で、操作の並びによって意味が生まれます。

**特徴：**

- 状態を“直接”扱う
- アルゴリズムや手順が主役
- コンピュータの実行モデルに最も近い

扱う対象が明確であればあるほど、手続き的思考は有効に働くかなと思います。

## オブジェクト指向：データと操作をひとまとまりにし、「誰に命ずるか」で考える

オブジェクト指向では、データ（状態）と操作（メソッド）を同じ場所にまとめ、そのまとまりを「主体」として扱うものかなと思います。
ここで中心になるのは"手順"ではなく、"誰に命ずるか"という視点です。

```go
order.Pay()
order.Cancel()
order.Ship()
```

これは「注文という"もの"がどう振る舞うべきか」というモデルに対して命じています。

**特徴：**

- 状態とふるまいを内包した"概念"を中心に捉える
- 実世界のメタファーに近く、理解しやすい

ビジネスロジックや振る舞いが複雑な場合、オブジェクト指向は特に適しているかなと思います。

## 関数型：データを入力してデータを出力する「変換」として捉える

関数型思考では、「データの変換」に焦点が当たるかなと思います。
状態を直接変更せず、新たな値を返すことで計算の結果を表現します。

```go
updated := Pay(order)
shipped := Ship(updated)
```

**特徴：**

- 入力→出力の関係だけを見る
- 状態は不変（immutable）で扱う
- 合成（compose）で複雑さを抑えられる

数学的でありながら、大規模システムでは予測可能性と安全性をもたらすアプローチかなと思います。

## 3つのパラダイムの違いは「どこに焦点を当てるか」

整理すると以下のようになります。

| パラダイム        | 視点（レンズ）       | 中心となるモデル      |
| ------------ | ------------- | ------------- |
| **手続き的**     | 操作の順序・手順      | 作業フロー         |
| **オブジェクト指向** | データ＋ふるまいのまとまり | メッセージング／概念モデル |
| **関数型**      | データの変換・合成     | 数学的写像         |

どれも同じ "データ操作" を行っていますが、見ている世界が異なります。

- 手続き的：How（どう実行するか）
- オブジェクト指向：Who（誰がやるか）
- 関数型：What（入力と出力の関係）

このレンズの違いが、コードの姿を変え、設計に影響を与えます。

## なぜこの区別が重要なのか

ひとつのパラダイムですべてを解決しようとすると、難しさが出てくるかもしれません。

- 手続き的に寄りすぎると、状態管理が煩雑になる
- オブジェクト指向に寄りすぎると、抽象化を誤って複雑性が増す
- 関数型に寄りすぎると、副作用の扱いが難しくなる

それぞれのパラダイムが得意とする領域があるので、適材適所で使い分けるのがよいかなと思います。

たとえば私解釈のクリーンアーキテクチャやDDDだと、

- ユースケース層は手続き的（フローを司る）
- ドメイン層はオブジェクト指向（概念モデルを表す）
- 計算部分や純粋な変換は関数型が適する

というハイブリッド構造になっています。

1つに決めるのではなく、 **複雑性に応じてレンズを切り替える** ことが重要かなと思います。

## まとめ：パラダイムは「考え方の道具箱」

手続き的、オブジェクト指向、関数型。
これらは対立するものではなく、複雑性に立ち向かうための"思考の道具"かなと思います。

- 手続き的は、手順と状態を扱うことが得意
- オブジェクト指向は、概念のまとまりを扱うのに向いている
- 関数型は、データ変換をシンプルにし、副作用を隔離する

データをどう見るかによって、設計のアプローチは大きく変わります。
だからこそ、複数のパラダイムを理解し、適切に選べることが、設計の質を高める重要なポイントかなと思います。
